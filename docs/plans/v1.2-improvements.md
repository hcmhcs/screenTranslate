# v1.2 개선 계획서

> 작성일: 2026-03-02
> 대상: ScreenTranslate v1.2
> 상태: 기획 (기술 검토 완료)

## 개요

v1.1 실사용에서 발견된 4가지 UX 개선사항을 정리한다.

| # | 이슈 | 심각도 | 영향 범위 |
|---|------|--------|----------|
| 1 | 번역 팝업 빈공간 과다 + 크기 조절 불가 | 중 | TranslationPopupView, TranslationPopupWindow |
| 2 | 히스토리 리스트 클릭 시 상세 보기 없음 | 중 | HistoryView |
| 3 | 원문 언어 피커에 언어팩 상태 미표시 | 하 | SettingsView |
| 4 | 메뉴바 단축키가 사용자 변경에 동기화되지 않음 | 중 | MenuBarView |

## 이슈 1: 번역 팝업 빈공간 과다 + 크기 조절 불가

### 현재 상태

스크린샷에서 확인되듯 번역 팝업 상단에 큰 빈 공간이 존재한다.

**근본 원인:**

`TranslationPopupWindow.calculateSize(for:)`에서 텍스트 높이를 추정할 때:

```swift
let sourceHeight = estimateTextHeight(result.sourceText)
let maxTextHeight = max(translatedHeight, translatedHeight + sourceHeight * 0.5)
let contentHeight = maxTextHeight + 100  // 패딩 + 버튼 + 토글 + 구분선
```

- 원문 높이의 50%를 **항상** 더하여 "원문 보기" 토글 시 여유 공간을 미리 확보한다.
- 원문이 길면 이 여유분이 과대 계산되어, 토글 전 상태에서 빈 공간으로 보인다.
- `maxHeight: 500`으로 제한하지만, 실제 필요한 높이보다 훨씬 큰 윈도우가 생성된다.
- `.frame(maxHeight: showingOriginal ? 140 : 220)` — ScrollView에 고정 maxHeight를 사용해, 텍스트가 적어도 220pt 공간을 차지한다.

**추가 문제:** 사용자가 팝업 크기를 수동으로 조절할 수 없다.

### 개선 방안

#### A. 동적 크기 계산 개선

**파일: `TranslationPopupWindow.swift`**

```
변경 전:
  let maxTextHeight = max(translatedHeight, translatedHeight + sourceHeight * 0.5)
  let contentHeight = maxTextHeight + 100

변경 후:
  // 원문 보기 토글 전에는 번역문 높이만 기준으로 계산
  let contentHeight = translatedHeight + 100  // 번역문 + 패딩/버튼/토글
```

- "원문 보기" 토글 시에는 `updateState()`가 호출되므로, 그때 원문 높이를 포함하여 재계산하면 된다.
- 다만 현재 `calculateSize()`는 `showingOriginal` 상태를 모른다.

**해결:** `TranslationPopupView` 내부의 `showingOriginal` 변경 시 외부에 크기 변경을 알리는 콜백 추가, 또는 윈도우가 SwiftUI의 intrinsic content size에 맞춰 자동으로 조절되도록 변경.

**권장 접근:** `calculateSize()` 계산 로직 개선 + `showingOriginal` 콜백 연동.

> **[기술 검토 주의사항]**
>
> `NSHostingView.intrinsicContentSize`는 ScrollView가 포함된 뷰에서 부정확할 수 있다.
> ScrollView는 가용 공간을 모두 차지하려 하므로, intrinsic size가 무한대 또는 과소 보고된다.
>
> **대신 `fittingSize`를 사용하거나**, ScrollView에 `.fixedSize(horizontal: false, vertical: true)`를
> 적용하여 콘텐츠에 맞는 자연 크기를 계산하게 한다.

```swift
// TranslationPopupView에서 고정 maxHeight 제거 + fixedSize 적용
ScrollView {
    Text(result.translatedText)
        .font(.body)
        .foregroundStyle(.primary)
        .textSelection(.enabled)
        .frame(maxWidth: .infinity, alignment: .leading)
}
.frame(maxHeight: 300)  // 상한만 유지, 최소값은 콘텐츠 기반
// 짧은 텍스트에서는 300보다 작게 표시, 긴 텍스트에서만 스크롤
```

```swift
// TranslationPopupWindow에서 calculateSize 개선
// 1단계: 번역문 높이만으로 기본 크기 계산 (원문 미포함)
let contentHeight = translatedHeight + 100
// 2단계: showingOriginal 콜백 수신 시 원문 높이 포함하여 재계산
```

**핵심:** `showingOriginal` 토글 시 윈도우 크기를 재조정하는 콜백 체인 추가.

```swift
// TranslationPopupView에 콜백 추가
let onToggleOriginal: (Bool) -> Void

// Toggle 변경 시 호출
Toggle("원문 보기", isOn: $showingOriginal)
    .onChange(of: showingOriginal) { _, newValue in
        onToggleOriginal(newValue)
    }
```

```swift
// TranslationPopupWindow에서 콜백 연결
// showingOriginal = true → 원문 높이를 포함한 크기로 애니메이션 리사이즈
// showingOriginal = false → 번역문만의 크기로 애니메이션 리사이즈
```

#### B. 팝업 크기 수동 조절 지원

**파일: `TranslationPopupWindow.swift`**

현재 `styleMask: [.borderless]`로 리사이즈 불가.

> **[기술 검토 주의사항]**
>
> `.borderless + .resizable`은 macOS에서 리사이즈 핸들이 보이지 않는 문제가 있다.
>
> `.titled` 추가 시 주의점:
> - 현재 `.regularMaterial` + `.clipShape(RoundedRectangle)` + `.shadow()` 커스텀 스타일과 충돌 가능
> - `.titled`의 시스템 타이틀바 영역이 SwiftUI 커스텀 배경과 겹침
> - `isMovableByWindowBackground = true`가 이미 설정됨 → 타이틀바 드래그와 중복
>
> **권장:** `.titled + .resizable + .fullSizeContentView`에 `titlebarAppearsTransparent = true` 적용.
> 단, SwiftUI 쪽 `.clipShape(RoundedRectangle)` + `.shadow()`는 제거하고
> NSPanel 레벨에서 `backgroundColor = .clear` + 커스텀 `contentView` 배경으로 전환해야
> 시각적 일관성이 유지된다.

```swift
// TranslationPopupWindow init()
super.init(
    contentRect: .zero,
    styleMask: [.titled, .resizable, .fullSizeContentView],
    backing: .buffered,
    defer: false
)
titleVisibility = .hidden
titlebarAppearsTransparent = true
// isMovableByWindowBackground는 .titled와 함께 자동 지원
```

**대안 (커스텀 스타일 유지 우선):**

현재 디자인을 최대한 유지하려면, `.borderless` 유지 + SwiftUI DragGesture 기반 리사이즈 핸들:

```swift
// TranslationPopupView 하단에 리사이즈 핸들 추가
HStack {
    Spacer()
    Image(systemName: "line.diagonal")
        .foregroundStyle(.tertiary)
        .gesture(DragGesture().onChanged { value in
            // NSWindow 크기 변경
        })
}
```

## 이슈 2: 히스토리 리스트 상세 보기

### 현재 상태

`HistoryView`에서 리스트 항목을 클릭해도 아무 반응이 없다. `selection: $selectedRecord`가 바인딩되어 있지만, 선택된 레코드에 대한 상세 화면이 구현되어 있지 않다. 번역문은 `lineLimit(2)`로 잘리고 원문도 `lineLimit(1)`로 표시되어, 긴 텍스트의 전체 내용을 확인할 수 없다.

### 개선 방안

히스토리 목록 옆에 상세 패널(detail pane)을 추가한다. macOS의 `NavigationSplitView` 또는 커스텀 마스터-디테일 레이아웃을 사용한다.

#### 상세 뷰 디자인

```
┌─────────────────────────────────────────────────────────┐
│  번역 히스토리                              [전체 삭제]  │
├───────────────────────┬─────────────────────────────────┤
│  리스트 (좌측)         │  상세 보기 (우측)                │
│                       │                                 │
│  ✅ English → Korean  │  ┌─ 번역문 ─────────────────┐   │
│  > Ralph Loop은 ...   │  │ Ralph Loop은 반복적이며   │   │
│                       │  │ 자기 참조적인 AI 개발     │   │
│  ✅ English → Korean  │  │ 루프를 가능하게 합니다.   │   │
│    랄프 루프           │  └─────────────────────────┘   │
│                       │                                 │
│  ✅ English → Korean  │  ┌─ 원문 ───────────────────┐   │
│    전체 로컬 환경을... │  │ Ralph Loop enables       │   │
│                       │  │ iterative, self-referential│  │
│                       │  │ AI development loops...    │  │
│                       │  └─────────────────────────┘   │
│                       │                                 │
│                       │  English → Korean  |  1분 35초  │
│                       │                                 │
│                       │  [원문 복사]  [번역문 복사]      │
└───────────────────────┴─────────────────────────────────┘
```

#### 구현 방식

> **[기술 검토 주의사항]**
>
> 1. **NSWindow 호스팅 환경**: HistoryView는 `WindowGroup`이 아닌 `NSWindow` +
>    `NSHostingView`에 호스팅된다. `NavigationSplitView`는 이 환경에서 정상 동작하지만,
>    윈도우 최소 크기를 충분히 확보해야 한다 (현재 600x500 → 적절함).
>
> 2. **@Model 선택 타입**: `TranslationRecord`는 SwiftData `@Model` (클래스).
>    `List(selection:)`의 selection 타입은 `TranslationRecord.ID?` (`UUID?`) 또는
>    `TranslationRecord?` 모두 가능. `@Model`은 `Identifiable`을 자동 준수하므로
>    `selection: $selectedRecord` (TranslationRecord?) 방식이 가장 단순.
>
> 3. **윈도우 크기 확장 고려**: 마스터-디테일 레이아웃에 맞게 기본 너비를 600→800으로 확대 권장.

**파일: `UI/History/HistoryView.swift`**

```swift
// NavigationSplitView 사용
NavigationSplitView {
    // 기존 List (마스터)
    List(historyManager.recentRecords, id: \.id, selection: $selectedRecord) {
        HistoryRowView(record: $0)
    }
} detail: {
    // 새로운 상세 뷰
    if let record = selectedRecord {
        HistoryDetailView(record: record)
    } else {
        Text("번역 기록을 선택하세요")
            .foregroundStyle(.secondary)
    }
}
```

**새 파일: `UI/History/HistoryDetailView.swift`**

```swift
struct HistoryDetailView: View {
    let record: TranslationRecord

    var body: some View {
        ScrollView {
            VStack(alignment: .leading, spacing: 16) {
                // 번역문 섹션
                Section("번역문") {
                    Text(record.translatedText ?? "")
                        .textSelection(.enabled)
                }

                Divider()

                // 원문 섹션
                Section("원문") {
                    Text(record.sourceText)
                        .textSelection(.enabled)
                        .foregroundStyle(.secondary)
                }

                Divider()

                // 메타 정보
                HStack {
                    // 언어 정보
                    // 타임스탬프
                }

                // 액션 버튼
                HStack {
                    Button("원문 복사") { ... }
                    Button("번역문 복사") { ... }
                }
            }
        }
    }
}
```

## 이슈 3: 원문 언어 피커 언어팩 상태 미표시

### 현재 상태

`SettingsView`에서 "번역 결과 언어" 피커에만 언어팩 상태 아이콘(설치됨/다운로드 필요/미지원)이 표시된다. "원문 언어" 피커에는 표시되지 않아, 사용자가 원문 언어의 언어팩 상태를 확인할 수 없다.

### 근본 원인

`LanguagePackManager.refreshStatuses(sourceCode:)`가 **원문 언어를 기준으로 각 타겟 언어의 상태**를 확인한다. 즉 상태는 `(source, target)` 쌍 기반이므로, 원문 피커에 표시하려면 **역방향** (각 원문 후보 → 현재 타겟) 상태를 추가로 확인해야 한다.

### 개선 방안

#### A. LanguagePackManager에 원문 기준 상태 추가

**파일: `Core/LanguagePackManager.swift`**

```swift
/// 타겟 언어 기준으로 각 원문 언어의 상태를 확인한다.
var sourceStatuses: [String: LanguageStatus] = [:]

func refreshSourceStatuses(targetCode: String) async {
    let targetLang = Locale.Language(identifier: targetCode)

    for lang in AppSettings.supportedLanguages {
        sourceStatuses[lang.code] = .checking
    }

    for lang in AppSettings.supportedLanguages {
        let source = Locale.Language(identifier: lang.code)
        let status = await availability.status(from: source, to: targetLang)
        switch status {
        case .installed: sourceStatuses[lang.code] = .installed
        case .supported: sourceStatuses[lang.code] = .available
        case .unsupported: sourceStatuses[lang.code] = .unsupported
        @unknown default: sourceStatuses[lang.code] = .unsupported
        }
    }
}
```

#### B. SettingsView 원문 피커에 아이콘 추가

**파일: `UI/Settings/SettingsView.swift`**

```swift
Picker("원문 언어", selection: $settings.sourceLanguageCode) {
    Text("자동 감지").tag("auto")
    Divider()
    ForEach(AppSettings.supportedLanguages, id: \.code) { lang in
        Label {
            Text(lang.name)
        } icon: {
            sourceStatusIcon(for: lang.code)  // 새로운 아이콘 함수
        }
        .tag(lang.code)
    }
}
```

#### C. 상태 갱신 타이밍

- `.task`: 초기 로드 시 양쪽 모두 갱신
- `.onChange(of: targetLanguageCode)`: 타겟 변경 시 원문 상태 재갱신
- `.onChange(of: sourceLanguageCode)`: 원문 변경 시 타겟 상태 재갱신 (기존)

### 수정 대상 파일

| 파일 | 변경 내용 |
|------|----------|
| `Core/LanguagePackManager.swift` | `sourceStatuses` 딕셔너리 + `refreshSourceStatuses()` 추가 |
| `UI/Settings/SettingsView.swift` | 원문 피커에 `sourceStatusIcon()` 추가, `.onChange(of: targetLanguageCode)` 연결 |

## 이슈 4: 메뉴바 단축키 동기화

### 현재 상태

`MenuBarView`에서 "번역하기" 버튼의 단축키가 하드코딩되어 있다:

```swift
Button("번역하기") {
    AppOrchestrator.shared.startTranslation()
}
.keyboardShortcut("T", modifiers: [.control, .shift])  // 항상 Ctrl+Shift+T
```

사용자가 설정에서 `KeyboardShortcuts.Recorder`를 통해 단축키를 변경해도, 메뉴바에 표시되는 단축키는 기본값(`Ctrl+Shift+T`)에서 변하지 않는다.

### 근본 원인

`.keyboardShortcut("T", modifiers: [.control, .shift])`는 SwiftUI의 정적 키보드 단축키로, `KeyboardShortcuts` 라이브러리와 연동되지 않는다.

### 개선 방안 — `.globalKeyboardShortcut()` 뷰 모디파이어 사용

> **[기술 검토 결과]** KeyboardShortcuts 라이브러리 소스 분석 결과,
> 이미 정확히 이 문제를 해결하는 API가 존재한다:
>
> `ViewModifiers.swift`에 정의된 `.globalKeyboardShortcut(_ name:)` (macOS 12.3+)
>
> 내부 동작:
> 1. `name.shortcut?.toSwiftUI`로 현재 단축키를 SwiftUI `KeyboardShortcut`으로 변환
> 2. `NotificationCenter` `.shortcutByNameDidChange` 구독 → 변경 시 자동 갱신
> 3. Recorder 활성 시 단축키 표시 자동 숨김 (충돌 방지)
> 4. 표시 전용 — 실제 동작은 글로벌 단축키(Carbon)가 처리

**수정 (1줄 변경):**

```swift
// 변경 전
Button("번역하기") {
    AppOrchestrator.shared.startTranslation()
}
.keyboardShortcut("T", modifiers: [.control, .shift])

// 변경 후
Button("번역하기") {
    AppOrchestrator.shared.startTranslation()
}
.globalKeyboardShortcut(.translate)
```

이 방식의 장점:
- **네이티브 메뉴 단축키 표시** — NSMenu 오른쪽에 단축키 기호(⌃⇧T) 표시 유지
- **자동 동기화** — 사용자가 설정에서 변경하면 메뉴바에 즉시 반영
- **Recorder 충돌 방지** — Recorder 활성화 시 메뉴의 단축키가 일시 해제
- **1줄 변경** — 기존 코드 구조 변경 불필요

### 수정 대상 파일

| 파일 | 변경 내용 |
|------|----------|
| `UI/MenuBar/MenuBarView.swift` | `.keyboardShortcut(...)` → `.globalKeyboardShortcut(.translate)` |

### 이슈별 검토 결과

#### 이슈 1: 팝업 빈공간 — 수정 필요

| 항목 | 원래 계획 | 검토 결과 |
|------|----------|----------|
| 크기 계산 | `intrinsicContentSize` 활용 | `intrinsicContentSize`는 ScrollView에서 부정확. **`fittingSize` 사용 또는 `fixedSize()` 적용** 필요 |
| 원문 토글 | calculateSize에서 미리 여유 확보 | **`onToggleOriginal` 콜백 → 동적 리사이즈** 패턴으로 변경 |
| 리사이즈 | `.titled + .resizable` | 가능하나 **기존 커스텀 스타일(.regularMaterial + clipShape + shadow)과 충돌 주의**. 스타일 전환 필요 |

#### 이슈 2: 히스토리 상세 — 문제 없음 (주의사항 추가)

| 항목 | 원래 계획 | 검토 결과 |
|------|----------|----------|
| NavigationSplitView | NSWindow + NSHostingView에서 사용 | **정상 동작 확인**. 윈도우 최소 크기 충분 |
| @Model 선택 바인딩 | `TranslationRecord?` | SwiftData @Model은 Identifiable 자동 준수. **문제 없음** |
| 윈도우 크기 | 600x500 | 마스터-디테일에 맞게 **800x500으로 확장 권장** |

#### 이슈 3: 원문 언어팩 — 문제 없음

| 항목 | 원래 계획 | 검토 결과 |
|------|----------|----------|
| 역방향 상태 확인 | `refreshSourceStatuses(targetCode:)` | **올바른 접근**. LanguageAvailability API는 (source, target) 쌍 기반이므로 역방향 확인 필수 |
